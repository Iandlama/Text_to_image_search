# README: Бенчмарк текстовых энкодеров для поиска мемов

Данный скрипт `benchmark.py` предназначен для объективного сравнения кандидатов на роль **текстового эмбеддера** во втором шаге пайплайна поиска мемов (архитектура **Вариант B**).  
Он прогоняет несколько моделей на реальном датасете мемов, вычисляет стандартные retrieval-метрики (nDCG@10, Recall@K, MRR) и оценивает устойчивость к деградации изображений, а также наличие паразитного кластера «плохого качества».

Все модели работают в полной изоляции - векторы разных моделей **никогда не смешиваются**, сравнение производится исключительно по скалярным метрикам. Это гарантирует корректность при разных размерностях и нормировках.

---

## 1. Требования и установка

### Необходимые зависимости

- Python 3.12 (строго)
- Основные библиотеки: `numpy`, `pillow`, `pyyaml`, `tqdm`, `pandas`, `matplotlib`
- Для OCR: `pytesseract` (и установленный Tesseract в системе)
- Для работы с моделями Hugging Face: `transformers==4.55.0`, `sentence-transformers`
- Локальный пакет `meme_embeddings` (должен быть в PYTHONPATH, либо рассположен около `benchmark.py`)

### Установка

```bash
pip install -r requirements.txt 
```

Убедитесь, что Tesseract установлен и доступен в PAT

## 2. Структура конфигурации (`models_config.yaml`)

Конфигурационный файл задаёт параметры датасета, аугментации, пути к кэшам и список моделей для бенчмарка. Пример полного файла приведён в `models_config.yaml`.

### Основные секции

| Секция | Описание |
|--------|----------|
| `dataset` | Пути к `metadata.jsonl` и папке с изображениями, расширение файлов, размер случайной выборки и seed. |
| `augmentation` | Параметры симуляции «плохого качества»: доля мемов, количество циклов JPEG-пересохранения, диапазон качества (quality_min/max), seed. |
| `cache_dir` | Корневая папка для кэшей OCR, деградированных картинок и эмбеддингов. |
| `output_dir` | Папка для сохранения результатов (CSV, Markdown, графики). |
| `batch_size` | Размер батча при кодировании текстов. |
| `models` | Список записей о моделях, каждая из которых содержит: `name`, `hf_id`, `embedder_type` (text/clip), `native_dim`, `target_dim`, префиксы для запросов/документов, `max_length` и др. |

### Формат записи модели

```yaml
- name: <уникальное имя модели>
  hf_id: <идентификатор на Hugging Face>
  embedder_type: text   # или clip
  native_dim: <исходная размерность>
  target_dim: <желаемая выходная размерность (≤ native_dim)>
  query_prefix: "..."   # опциональный префикс для запросов
  passage_prefix: "..." # опциональный префикс для документов
  max_length: <максимальная длина в токенах>
  trust_remote_code: true/false
  notes: ...
```

Если `target_dim` меньше `native_dim`, используется **проекция** (по умолчанию - усечение первых `target_dim` координат). Для моделей с Matryoshka Representation Learning (MRL) такое усечение является штатным режимом; для остальных - допустимым компромиссом.

## 3. Модуль OCR (`my_ocr.py`)

В текущей реализации `my_ocr.py` представляет собой тестовый ocr на основе Tesseract:

```python
import pytesseract
from PIL import Image

def get_ocr(img: Image.Image) -> str:
    return pytesseract.image_to_string(img)
```

Эта функция вызывается для каждой картинки мема (оригинальной и деградированной). Вы можете заменить её на более продвинутое решение (например, на базе TrOCR, EasyOCR или коммерческого API) - главное, чтобы она принимала PIL.Image и возвращала строку.

## 4. Переменные окружения (файл `.env`)

Бенчмарк использует библиотеку `python-dotenv` для загрузки переменных из файла `.env`. Важные переменные:

- `HF_TOKEN=...` - **обязателен** для моделей, требующих аутентификации (например, `google/embeddinggemma-300m`). Без него загрузка таких моделей завершится ошибкой.
- `HF_HOME=...` (опционально) - путь к кэшу Hugging Face, куда будут сохраняться загруженные веса.
- `LOG_LEVEL=NONE` (или `INFO`, `DEBUG`) - управляет детализацией логирования.

**Рекомендация:** скопируйте `.env.example` в `.env` и вставьте свой Hugging Face токен перед запуском.

## 5. Запуск бенчмарка

### Базовый запуск

```bash
python benchmark.py --config models_config.yaml
```

Будут протестированы **все модели** из конфига. Результаты сохранятся в `output_dir`.

### Выборочный запуск

```bash
python benchmark.py --config models_config.yaml --models bge-m3 multilingual-e5-base
```

Будут обработаны только указанные модели (по полю `name`).

### Экспорт конфига одной модели

```bash
python benchmark.py --config models_config.yaml --export-model bge-m3 --export-path chosen_model.yaml
```

Создаст файл `chosen_model.yaml`, содержащий только запись этой модели, удобно для дальнейшего использования в основном пайплайне.

## 6. Этапы работы бенчмарка

Процесс прогона одной модели включает следующие шаги:

### 6.1 Загрузка датасета
Из `metadata.jsonl` и папки `images` берётся случайная выборка заданного размера. Каждый элемент содержит поля: `id`, `title`, `image_desc`, `meaning`.

### 6.2 Формирование запросов и документов
- **Запрос (query)** = поле `meaning` (смысл мема). Это максимально приближено к тому, как пользователь будет формулировать поисковый запрос.
- **Документ (doc)** = `title` + `image_desc` + OCR-текст. Именно этот текст будет индексироваться в продуктивной системе.
- **Релевантность**: для каждого запроса `i` релевантным считается документ `i` (self-id). Это простой, но надёжный способ получить ground truth без ручной разметки.

### 6.3 Оптическое распознавание текста (OCR)
Для каждого мема выполняется OCR на оригинальном изображении. Результаты кэшируются в `cache_dir/ocr_cache.json`.

### 6.4 Симуляция «плохого качества»
Часть мемов (доля `augmentation.fraction`) подвергается многократному пересохранению в JPEG с низким случайным качеством и ресайзом. Этот процесс имитирует деградацию при репостах в соцсетях. Для каждого такого мема генерируется деградированная версия картинки (сохраняется в кэше) и заново выполняется OCR.

### 6.5 Кодирование текстов
Для каждой модели:
- Загружается эмбеддер (`create_embedder` из пакета `meme_embeddings`).
- Кодируются все запросы и документы (чистые и «смешанные», где для деградированных мемов используется их перераспознанный OCR).
- Векторы сохраняются в `.npy`-файлы под кэш-директорией модели, что позволяет избежать повторного кодирования при перезапуске.

### 6.6 Расчёт метрик
Вычисляется матрица косинусной близости между запросами и документами (все векторы нормализованы). На основе рангов релевантных документов считаются:
- `Recall@K` (K = 1, 5, 10)
- `MRR` (Mean Reciprocal Rank)
- `nDCG@10`
- `mean_rank`

Эти метрики вычисляются для:
- **чистого корпуса** (все документы по оригинальному OCR)
- **подвыборки деградированных мемов** на чистом корпусе (baseline)
- **той же подвыборки** на «смешанном» корпусе (деградированные документы с новым OCR) - чтобы оценить, насколько ухудшается качество поиска именно для этих мемов.

### 6.7 Оценка кластерной утечки (`cluster_leakage_score`)
Для выявления паразитного кластера мемов с низким качеством вычисляется разница между средней косинусной близостью между деградированными документами и средней близостью между случайной выборкой чистых документов. **Положительное значение** указывает на то, что деградированные мемы «слипаются» в эмбеддинг-пространстве сильнее, чем обычные, - признак нежелательного артефакта.

## 7. Результаты

После завершения прогона всех моделей в `output_dir` создаются:

- `benchmark_results.csv` - таблица со всеми метриками для каждой модели.
- `benchmark_results.md` - та же таблица в формате Markdown.
- Графики (PNG):
  - `ndcg10_comparison.png` - сравнение nDCG@10 на чистом корпусе.
  - `recall_comparison.png` - Recall@1, @5, @10.
  - `degradation_robustness.png` - nDCG@10 на подвыборке деградированных мемов: baseline vs mixed.
  - `cluster_leakage.png` - кластерная утечка (красные столбцы - высокий риск).
  - `dim_vs_quality.png` - зависимость качества от размерности вектора.

Итоговую таблицу можно просмотреть в терминале - она выводится автоматически.

---

## 8. Расширение и кастомизация

- Добавление новой модели: просто допишите её запись в `models_config.yaml`, указав корректный `hf_id` и параметры.
- Замена OCR: отредактируйте `my_ocr.py` - функция должна принимать PIL.Image и возвращать строку.
- Изменение метрик: в функции `retrieval_metrics` можно добавить другие K или варианты nDCG.
